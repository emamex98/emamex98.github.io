Glint Components

So, you've heard about the Glimmer Framework, right? If not, don't worry if you haven't... GimmerJS, as described on its website is a "fast and light-weight UI components framework" for web applications, developed by the same team behind EmberJS.

Glimmer components are based on the concept of virtual DOM. The virtual DOM is a representation of the DOM that is stored in memory. When a Glimmer component changes, the virtual DOM is updated, and then the browser updates the actual DOM to reflect the changes in the virtual DOM. This approach to rendering is much faster than traditional DOM rendering, because it only updates the parts of the DOM that have actually changed.

Now, Glint is described on its website as "a set of tools to aid in developing code that uses the Glimmer VM for rendering... to provide feedback and enforce correctness both locally during editing and project-wide in CI". In short, Glint components are a way to create reusable and testable UI elements based on the Glimmer framework and that take advantage of its virtual DOM to do so.

Here are some of the benefits of using Glint components:

  • Reusability: Glint components are reusable, meaning they can be used in multiple places in the UI. This can save time and effort for developers.
  • Testability: Glint components are testable, which means that developers can easily test their behavior and receive instant feedback if using a compatible editor or through its CLI tool at the moment of type-cheking.
  • Ease of use: Glint components are easy to use. In the next secction, an sample implmentation will show how these components can be easily implemented in frameworks such as EmberJS.

Overall, monorepos can be a good choice for projects or organizations that need to improve code reuse, collaboration, testability, and tooling.


Implementing Glint in Ember JS

Since the Glimmer framework and glint were both created by the EmberJS team, it makes sense to show a sample implementation in Ember. Here's a sample TS file for an ember component: 

                
import Component from '@ember/component';
import { computed } from '@ember/object';

export interface GreetingSignature {
    Args: {
        message: string;
        target?: string;
    };
    Blocks: {
        default: [greeting: string];
    };
}

// We define this type alias so that we can extend it below:
type GreetingArgs = GreetingSignature['Args'];

// This line declares that our component's args will 
// be 'splatted' on to the instance:
export default interface Greeting extends GreetingArgs {}

export default class Greeting extends Component {
    @computed('target')
    private get greetingTarget() {
        // Therefore making `this.target` a legal  
        // `string | undefined` property access:
        return this.target ?? 'World';
    }
}

And the component's template .hbs file would be as simple as:

                
{{yield (concat @message ', ' this.greetingTarget '!')}}
